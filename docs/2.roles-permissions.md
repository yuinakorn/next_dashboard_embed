# Roles And Permissions

เอกสารนี้กำหนดบทบาทผู้ใช้, ขอบเขตสิทธิ์ และแนวทางรับ SSO/JWT สำหรับ Governed Dashboard Portal

## Auth Direction

ระบบจริงจะไม่ทำ identity provider เอง แต่จะรับ SSO/JWT จากระบบ auth หลักของหน่วยงาน

JWT ควรมีข้อมูลขั้นต่ำ:

- user id
- display name
- title/position
- department
- team id
- roles
- category scopes
- token issue/expire time

ตัวอย่าง payload ที่ mock ไว้ในระบบ:

```json
{
  "sub": "u-001",
  "name": "แอดมิน Yui",
  "title": "Project Manager",
  "department": "Digital Health Strategy",
  "team_id": "team-digital-health",
  "roles": ["project_manager", "editor", "viewer"],
  "scopes": [
    {
      "teamId": "team-digital-health",
      "categoryIds": ["digital-health", "digital-health-sso", "digital-health-data"]
    }
  ],
  "iat": 1777885200,
  "exp": 1777971600
}
```

## Roles

### system_admin

ผู้ดูแลระบบทั้งหมด

สิทธิ์หลัก:

- สร้าง category หลัก
- แก้ไข category ทุกระดับ
- สร้าง/แก้ไข dashboard ทุกตัว
- approve/publish/archive dashboard
- pin dashboard
- อ่าน audit log

### category_admin

ผู้ดูแล category หลักที่ได้รับมอบหมาย

สิทธิ์หลัก:

- สร้าง sub-category ใต้ category ที่อยู่ใน scope
- แก้ไข category ใน scope
- สร้าง/แก้ไข dashboard ใน scope
- approve/publish/archive dashboard ใน scope
- pin dashboard ใน scope
- อ่าน audit log ของ scope

### project_manager

ผู้ดูแลพื้นที่ของโครงการหรือทีม

สิทธิ์หลัก:

- สร้าง sub-category ใต้ category ที่ตนมีสิทธิ์
- สร้าง dashboard ใน category ที่อยู่ใน scope
- แก้ไข dashboard ของทีมตัวเอง
- ส่ง dashboard เข้า review
- archive dashboard ของทีมตัวเอง

### editor

ผู้สร้างหรือแก้ไข dashboard

สิทธิ์หลัก:

- สร้าง dashboard ใน category ที่อยู่ใน scope
- แก้ไข dashboard ที่ตนเองหรือทีมเป็นเจ้าของ
- ส่ง dashboard เข้า review

### viewer

ผู้ใช้งานทั่วไปภายในระบบ

สิทธิ์หลัก:

- ดู dashboard ที่ published และมีสิทธิ์เข้าถึง
- favorite dashboard
- เปิด external link fallback

## Scope Model

สิทธิ์ไม่ควรผูกกับ role อย่างเดียว เพราะ project manager หรือ category admin แต่ละคนควรเห็นพื้นที่ไม่เท่ากัน

จึงใช้แนวคิด scope:

```ts
type TeamScope = {
  teamId: string;
  categoryIds: string[];
};
```

ตัวอย่าง:

- project manager ทีม Digital Health เห็นเฉพาะ `digital-health`
- category admin งานบริการเห็นเฉพาะ `operations`
- system admin ไม่ต้องอาศัย scope เพราะมีสิทธิ์ global

## Visibility Rules

กติกาการมองเห็น dashboard ขั้นต้น:

- dashboard ที่ `published` และ `public` เห็นได้จากหน้า public โดยไม่ต้อง login
- dashboard ที่ `published` และ `internal` เห็นได้ใน portal หลัง login
- dashboard ที่ `confidential` หรือ `restricted` ควรตรวจสิทธิ์จาก scope/role เพิ่มเติม
- dashboard ที่ยังไม่ published เห็นได้เฉพาะ owner team หรือผู้มีสิทธิ์ review/publish

## Next Decisions

เรื่องที่ต้องตัดสินใจก่อนต่อ SSO จริง:

- JWT จะส่งมาทาง cookie หรือ Authorization header
- role/permission จะอยู่ใน JWT ทั้งหมด หรือ portal ต้อง query เพิ่มจาก DB
- category scope จะมาจาก auth หลัก หรือกำหนดใน portal
- token refresh และ session timeout จะจัดการที่ระบบไหน
- ต้องรองรับ user หลายทีมพร้อมกันหรือไม่
