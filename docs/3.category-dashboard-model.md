# Category And Dashboard Model

เอกสารนี้กำหนด model สำหรับ category, dashboard catalog, metadata และ schema ที่ควรใช้เมื่อเริ่มต่อฐานข้อมูลจริง

## Category Model

category เป็นโครงสร้างแบบ tree

- admin สร้าง root category
- category admin/project manager สร้าง sub-category ภายใต้ root category ที่ตนมี scope
- category แต่ละรายการมี owner team
- category ไม่ควรถูกลบจริงถ้ามี dashboard อยู่ ควรใช้ archived/inactive status

ข้อมูลขั้นต่ำ:

| Field | Type | Note |
| --- | --- | --- |
| `id` | string | primary key |
| `name` | string | ชื่อ category |
| `parent_id` | string/null | null คือ root category |
| `owner_team_id` | string | team ที่ดูแล |
| `status` | string | active/inactive/archived |
| `sort_order` | number | เรียงลำดับใน sidebar |
| `created_by` | string | user id |
| `created_at` | datetime | เวลาสร้าง |
| `updated_at` | datetime | เวลาแก้ไขล่าสุด |

## Dashboard Model

dashboard คือรายการ catalog ที่ชี้ไปยัง embed/external dashboard ภายนอก

ข้อมูลขั้นต่ำ:

| Field | Type | Note |
| --- | --- | --- |
| `id` | string | primary key |
| `title` | string | ชื่อ dashboard |
| `description` | text | คำอธิบาย |
| `provider` | string | Looker Studio, Superset, Grafana, Metabase, Power BI, Custom |
| `category_id` | string | category หลักที่ dashboard อยู่ |
| `owner_team_id` | string | team เจ้าของ |
| `owner_user_id` | string | user เจ้าของหลัก |
| `status` | string | draft/in_review/published/rejected/archived |
| `sensitivity` | string | public/internal/confidential/restricted |
| `embed_url` | text | URL สำหรับ iframe/embed |
| `external_url` | text | fallback URL |
| `refresh_frequency` | string | daily/weekly/monthly/manual/unknown |
| `data_source_note` | text | อธิบายแหล่งข้อมูล |
| `last_reviewed_at` | datetime/null | review ล่าสุด |
| `published_at` | datetime/null | เผยแพร่ล่าสุด |
| `created_by` | string | user id |
| `created_at` | datetime | เวลาสร้าง |
| `updated_at` | datetime | เวลาแก้ไขล่าสุด |

## Metadata

metadata ที่ควรใช้เพื่อให้ระบบเป็น governed portal จริง:

- owner
- owner team
- category
- provider
- status
- sensitivity
- tag
- data source note
- refresh frequency
- last reviewed date
- published date
- usage count
- embed health

## Tags

ควรแยก tag เป็นตารางกลาง ไม่เก็บเป็น string เดียว เพื่อค้นหาและ filter ได้ดี

ตารางที่ควรมี:

- `tags`
- `dashboard_tags`

## Favorites

favorite เป็นความสัมพันธ์ระหว่าง user กับ dashboard

ข้อมูลขั้นต่ำ:

| Field | Type |
| --- | --- |
| `user_id` | string |
| `dashboard_id` | string |
| `created_at` | datetime |

## Permissions

permission มีได้ 2 ชั้น:

- role permission: สิทธิ์ตามบทบาท เช่น `dashboard:create`
- scope permission: สิทธิ์ตาม category/team เช่น user นี้สร้าง dashboard ได้ใน category ไหน

ถ้าต้องทำละเอียดขึ้นในอนาคต สามารถเพิ่มตาราง:

- `category_permissions`
- `dashboard_permissions`

## Audit Logs

ระบบนี้ควรมี audit log ตั้งแต่ MVP เพราะแกนหลักคือ governance

action ที่ควรบันทึก:

- create category
- update category
- create dashboard
- update dashboard
- submit review
- approve/reject
- publish/unpublish
- archive
- pin/unpin
- change sensitivity
- change embed URL

ข้อมูลขั้นต่ำ:

| Field | Type | Note |
| --- | --- | --- |
| `id` | string | primary key |
| `actor_user_id` | string | ผู้ทำรายการ |
| `action` | string | action key |
| `entity_type` | string | category/dashboard/permission |
| `entity_id` | string | id ของ entity |
| `before_json` | json/null | ค่าเดิม |
| `after_json` | json/null | ค่าใหม่ |
| `created_at` | datetime | เวลาที่เกิด action |

## Public Visibility Rule

หน้า public ควร query เฉพาะ:

```sql
status = 'published'
AND sensitivity = 'public'
```

ห้ามใช้ rule จาก frontend อย่างเดียว ต้อง enforce ที่ query/API ด้วย

## Internal Visibility Rule

หน้า internal portal ควรใช้ logic:

- system admin เห็นทั้งหมด
- published/internal เห็นได้หลัง login ตาม policy หน่วยงาน
- confidential/restricted ต้องตรวจ role/scope
- draft/in_review/rejected เห็นเฉพาะ owner team หรือ reviewer
