# MVP Implementation Plan And Handoff

เอกสารนี้ใช้เป็น handoff สำหรับการสนทนาถัดไป เพื่อให้ทำงานต่อจากสถานะปัจจุบันได้ทันทีโดยไม่ต้องไล่บริบทใหม่ทั้งหมด

## สถานะล่าสุด

ระบบถูกขยับจาก mock UI ไปสู่ MVP ที่อ่านและเขียนข้อมูลจริงบางส่วนผ่านฐานข้อมูลแล้ว หน้า UI หลักถูกปรับให้เป็นทิศทางเดียวกันตามแนว Governed Dashboard Portal และมี public route, catalog, review queue, audit log, viewer และ create form ที่ใช้งานกับข้อมูลจริงมากขึ้น

current user ถูก centralize แล้วผ่าน `src/lib/auth/current-user.ts` ทุกหน้าและ API หลักเรียก `getCurrentUser()` แทนการ import `mockCurrentUser` โดยตรง

สิ่งที่ยืนยันแล้ว:

- `./node_modules/.bin/eslint` ผ่าน
- `./node_modules/.bin/tsc --noEmit` ผ่าน
- `npm run build` ผ่านแล้ว หลังเพิ่ม edit flow
- route หลักเปิดใน in-app browser ได้โดยไม่มี console error หรือ warning
- ทดสอบ approve dashboard ใน `/review` แล้วเขียนสถานะลงฐานข้อมูลและสร้าง audit log ได้จริง
- เพิ่ม edit dashboard flow แล้ว:
  - `src/app/dashboards/[id]/edit/page.tsx`
  - `PATCH /api/dashboards/[id]`
  - preload metadata จาก DB
  - update metadata/embed/category/tag/refesh/data source note
  - permission check ผ่าน `canUpdateDashboard()`
  - audit log action `dashboard.update`
- เพิ่ม lifecycle workflow แล้ว:
  - `POST /api/dashboards/[id]/lifecycle`
  - submit draft/rejected เป็น `in_review`
  - archive dashboard พร้อม audit log action `dashboard.archive`
  - review API บังคับให้ approve/reject ได้เฉพาะรายการ `in_review`
  - reject ต้องระบุ note
  - หน้า `/dashboards/[id]` มี lifecycle panel และ audit trail ล่าสุดของ dashboard นั้น
- เพิ่ม search/filter จริงแล้ว:
  - `/catalog` ค้นหาด้วยชื่อ/คำอธิบาย/tag/owner/provider/category/status/sensitivity
  - `/catalog` filter provider/status/sensitivity และ sort updated/views/title
  - `/public` ค้นหา public dashboard ด้วยชื่อ/คำอธิบาย/tag/provider/category
  - `/public` filter provider/category และ sort views/updated/title
  - `/audit` ค้นหาด้วย action/entity/title/note/actor
  - `/audit` filter action/entity/date range

ข้อจำกัดที่ยังมี:

- auth ยังไม่ใช่ SSO/session จริง แต่มี current-user adapter กลางแล้ว และ dev mode ยัง fallback ไปที่ mock user อย่างตั้งใจ
- CRUD ยังไม่ครบ มี create, edit, submit/review/publish/reject/archive แล้ว แต่ยังไม่มี delete แบบถาวร และยังไม่มี restore จาก archived
- ยังไม่มีหน้า user/permission management สำหรับจัดการผู้ใช้ role team และ scope จริง
- ยังไม่มีหน้า category management สำหรับจัดการโครงสร้างหมวดหมู่จริง

### Auth Integration

เพิ่ม `src/lib/auth/current-user.ts` เป็น current user adapter กลางแล้ว

พฤติกรรมปัจจุบัน:

- default ใน development ใช้ `PORTAL_AUTH_MODE=dev` และ fallback ไปที่ `getMockCurrentUser()`
- default ใน production ใช้ `PORTAL_AUTH_MODE=trusted-header`
- trusted-header mode อ่าน header `x-portal-user`
- ค่า `x-portal-user` ต้องเป็น base64url JSON payload ใน shape เดียวกับ `MockJwtPayload`
- ถ้า production ไม่มี authenticated user จะ throw `Missing authenticated portal user.`

route/page/API ที่เปลี่ยนมาใช้ `getCurrentUser()` แล้ว:

- `src/app/page.tsx`
- `src/app/catalog/page.tsx`
- `src/app/review/page.tsx`
- `src/app/audit/page.tsx`
- `src/app/dashboards/[id]/page.tsx`
- `src/app/dashboards/new/page.tsx`
- `src/app/api/dashboards/route.ts`
- `src/app/api/dashboards/[id]/review/route.ts`

## จุดที่ต้องรู้ก่อนทำต่อ

การทดสอบ review workflow ได้เปลี่ยน seed row จริงใน local DB:

- dashboard `db-004` จากเดิม `in_review`
- ถูก approve เป็น `published`
- มี audit log action `dashboard.publish`

ถ้าต้องการให้ review queue มี item สำหรับทดสอบอีกครั้ง ให้เพิ่ม seed ใหม่หรือ reset row นี้กลับเป็น `in_review` ด้วยคำสั่ง SQL ที่ตั้งใจทำเท่านั้น อย่า reset DB ทั้งก้อนถ้าไม่จำเป็น

## Priority ถัดไป

### 1. ต่อ Auth/Current User เข้ากับของจริง

ตอนนี้ระบบมี service กลาง `src/lib/auth/current-user.ts` แล้ว แต่วิธีได้ user จริงยังเป็น adapter contract ไม่ใช่ SSO/session จริง งานถัดไปคือผูก adapter นี้เข้ากับระบบ auth ที่ใช้จริง

เป้าหมายขั้นต่ำ:

- เลือก production auth source จริง เช่น SSO callback, JWT, reverse proxy trusted header หรือ session cookie
- map user จริงเข้ากับ role/team/category scope จาก DB
- ทำ unauthenticated response ให้เหมาะกับ page และ API
- คง dev fallback ไว้ แต่ห้ามให้ production fallback ไป mock โดยเงียบ

ผลลัพธ์ที่ควรได้:

- `/catalog`, `/review`, `/dashboards/new`, API review และ API create ใช้ user source จริงจาก adapter เดียวกัน
- permission model เริ่มผูกกับ user จริง ไม่ใช่ mock

### 2. ทำ User / Permission Management

ตอนนี้ workflow และ search/filter หลักพร้อมแล้ว แต่ production ยังขาดหน้าจัดการสิทธิ์ที่เป็นแกนของ governed portal

ควรเพิ่ม:

- หน้า `/admin/users` หรือ `/settings/users`
- list users จาก DB หรือ source ที่ auth map มา
- assign role/team/category scope
- audit log เมื่อเปลี่ยน role/scope
- permission guard เฉพาะ `system_admin` หรือ role ที่เหมาะสม

### 3. ทำ Category Management

ควรเพิ่ม:

- หน้า `/categories` หรือ `/admin/categories`
- แสดง tree หมวดหมู่
- create root/child category ตาม permission
- edit owner team และชื่อหมวดหมู่
- audit log เมื่อสร้าง/แก้ไข category

### 4. Build และ Deploy Readiness

ก่อนถือว่าใช้งานจริงได้ ต้องแก้ `npm run build` ให้ผ่านในเครื่องและ CI

สถานะล่าสุด:

- `npm run build` ผ่านในเครื่องแล้ว
- ยังควรตรวจซ้ำใน CI/deploy environment เพราะ handoff ก่อนหน้ามีประวัติติด `lightningcss` native binary

ควรทำ:

- รัน `npm run build`
- ถ้า CI ยังติด ให้ตรวจ Node/npm architecture และ native package ที่เกี่ยวข้องกับ `lightningcss`

## Definition Of Done สำหรับ MVP ใช้งานจริง

ถือว่า MVP พร้อมใช้งานจริงเมื่อ:

- auth/current user ไม่ใช่ mock
- user เห็น dashboard ตาม permission จริง
- admin/editor สร้างและแก้ dashboard ได้
- submit/review/publish/reject/archive มี audit log
- public page แสดงเฉพาะ dashboard ที่ public และ published
- catalog ค้นหาและ filter ได้จริง
- embed fallback ทำงานเมื่อ iframe ใช้ไม่ได้
- `npm run lint`, `tsc --noEmit`, และ `npm run build` ผ่าน
- มี seed/migration ที่รันซ้ำได้อย่างคาดเดาได้
