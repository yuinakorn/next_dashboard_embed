# Governed Dashboard Portal - Project Phases

## Vision

ระบบนี้คือศูนย์กลางบริหารจัดการ dashboard แบบ embed จากระบบภายนอก เช่น Looker Studio, Superset, Metabase, Grafana, Power BI, Tableau หรือแหล่งอื่น ๆ โดยระบบของเราทำหน้าที่เป็น portal สำหรับจัดหมวดหมู่, กำหนดสิทธิ์, ค้นหา, ตรวจสอบ, อนุมัติ และควบคุมคุณภาพของ dashboard ภายในองค์กร

เป้าหมายหลักไม่ใช่การสร้างกราฟเอง แต่คือการทำให้ dashboard ที่กระจายอยู่หลายระบบกลายเป็นสินทรัพย์ข้อมูลที่ค้นหาได้, มีเจ้าของชัดเจน, มีสิทธิ์ควบคุม, มีสถานะเผยแพร่ และมี governance ที่ตรวจสอบย้อนหลังได้

## Core Direction

- เน้นแนวทาง Governed Dashboard Portal ก่อน
- ใช้แนวคิด Dashboard Marketplace เฉพาะบางส่วนที่มีประโยชน์กับ MVP เช่น dashboard ยอดนิยม, ล่าสุด, ของทีมฉัน, และ pin โดยผู้บริหาร
- รองรับ embed จากหลายแหล่ง โดยไม่จำกัดเฉพาะ platform ใด platform หนึ่ง
- แยกสิทธิ์ของระบบ portal ออกจากสิทธิ์ของเครื่องมือ dashboard ภายนอกอย่างชัดเจน
- ต้องมี fallback เป็น external link เสมอเมื่อ embed ไม่ได้

## Phase 0 - Product Framing

เป้าหมาย: ตกลงขอบเขตของระบบให้ชัดก่อนเริ่มพัฒนา

งานหลัก:

- นิยามชื่อระบบและเป้าหมายหลัก
- ระบุประเภทผู้ใช้งานหลัก เช่น admin, project manager, viewer
- ระบุ dashboard provider ที่ต้องรองรับในช่วงแรก
- ตกลงว่า dashboard ที่เพิ่มเข้ามาเป็น embed, external link หรือทั้งสองแบบ
- ตกลง policy เบื้องต้นเรื่อง domain ที่อนุญาตให้ embed
- ระบุข้อจำกัดสำคัญ เช่น private dashboard, SSO, CSP, X-Frame-Options

ผลลัพธ์ที่ควรได้:

- Product scope สำหรับ MVP
- รายการ role ขั้นต้น
- รายการ provider ขั้นต้น
- ข้อกำหนด security ขั้นต้นสำหรับ iframe/embed

## Phase 1 - Foundation And Access Control

เป้าหมาย: สร้างฐานระบบผู้ใช้, role, permission และ category หลัก

ฟีเจอร์หลัก:

- Authentication
- Role-based access control
- Admin สร้าง category หลัก
- Project manager สร้าง sub-category ภายใต้ category ที่ตนเองมีสิทธิ์
- กำหนด owner ของ category และ dashboard
- กำหนดสิทธิ์ระดับ category
- Audit log สำหรับ action สำคัญ

บทบาทตัวอย่าง:

- `system_admin` จัดการระบบทั้งหมด
- `category_admin` ดูแล category หลักที่ได้รับมอบหมาย
- `project_manager` สร้าง sub-category และ dashboard ในพื้นที่ของตัวเอง
- `editor` เพิ่มหรือแก้ไข dashboard ตามสิทธิ์
- `viewer` เปิดดู dashboard ที่ได้รับอนุญาต

ผลลัพธ์ที่ควรได้:

- ผู้ใช้เข้าใช้งานได้
- ระบบมี category tree
- สิทธิ์พื้นฐานทำงานได้
- การสร้าง/แก้ไขข้อมูลสำคัญถูกบันทึกใน audit log

## Phase 2 - Dashboard Catalog MVP

เป้าหมาย: ให้ผู้ใช้สร้างและจัดการ dashboard embed ได้จริง

ฟีเจอร์หลัก:

- CRUD dashboard
- เพิ่ม embed URL หรือ external URL
- เลือก provider type เช่น Looker Studio, Superset, Grafana, Metabase, Power BI, Tableau, Custom
- Preview iframe
- External link fallback
- Dashboard metadata
- Tag
- Search และ filter ขั้นต้น
- Favorite dashboard

Metadata ที่ควรมี:

- ชื่อ dashboard
- คำอธิบาย
- category และ sub-category
- owner
- provider
- embed URL
- external URL
- tag
- data source note
- refresh frequency
- sensitivity level
- status

ผลลัพธ์ที่ควรได้:

- ผู้ใช้เพิ่ม dashboard ได้
- ผู้ใช้ดู dashboard ผ่าน portal ได้
- ค้นหา dashboard ได้
- เปิด dashboard แบบ external link ได้เมื่อ embed ไม่สำเร็จ

## Phase 3 - Governance Workflow

เป้าหมาย: เพิ่มการควบคุมคุณภาพและสถานะการเผยแพร่ dashboard

Workflow หลัก:

- `draft` dashboard ยังไม่เผยแพร่
- `in_review` ส่งให้ผู้มีสิทธิ์ตรวจสอบ
- `published` เผยแพร่ให้ผู้ใช้ตามสิทธิ์
- `rejected` ถูกตีกลับพร้อมเหตุผล
- `archived` เลิกใช้งาน แต่ยังเก็บประวัติ

ฟีเจอร์หลัก:

- Submit for review
- Approve / reject
- Publish / unpublish
- Archive
- Review comment
- Change history
- Audit log แบบละเอียด
- Verified badge สำหรับ dashboard ที่ผ่านการตรวจสอบ

ผลลัพธ์ที่ควรได้:

- dashboard สำคัญไม่ถูกเผยแพร่โดยไม่มีการตรวจสอบ
- มีประวัติว่าใครแก้ไขหรืออนุมัติอะไร
- ลด dashboard ซ้ำ, dashboard ผิด, dashboard ไม่มีเจ้าของ

## Phase 4 - Home And Discovery

เป้าหมาย: ทำให้ผู้ใช้ค้นพบ dashboard ที่ควรใช้งานได้ง่ายตั้งแต่หน้าแรก

หน้า Home ควรมี:

- Dashboard ที่ถูก pin โดยผู้บริหารหรือ admin
- Dashboard ยอดนิยม
- Dashboard ล่าสุด
- Dashboard ของทีมฉัน
- Dashboard ที่ฉัน favorite
- Dashboard ที่เพิ่งถูก publish
- Dashboard ที่ใกล้หมดอายุหรือควร review ใหม่

ฟีเจอร์หลัก:

- Pin dashboard
- Track view count
- Track recent activity
- Team-based dashboard section
- Favorite section
- Quick search
- Filter ตาม category, tag, provider, owner, status

ผลลัพธ์ที่ควรได้:

- ผู้ใช้ไม่ต้องจำว่า dashboard อยู่ที่ไหน
- dashboard สำคัญถูกดันขึ้นหน้าแรก
- dashboard ที่มีการใช้งานจริงถูกมองเห็นง่ายขึ้น

## Phase 5 - Security And Embed Hardening

เป้าหมาย: ลดความเสี่ยงจาก iframe และ external content

ฟีเจอร์หลัก:

- Allowlist domain สำหรับ embed
- Blocklist domain ที่ไม่อนุญาต
- iframe sandbox policy
- ตรวจสอบ URL format
- แยก embed URL กับ external URL
- แสดงคำเตือนเมื่อ provider ไม่สามารถ embed ได้
- บันทึก embed failure
- กำหนด sensitivity level ของ dashboard

ข้อควรระวัง:

- สิทธิ์ใน portal ไม่ได้แทนสิทธิ์ใน Looker Studio, Superset หรือ provider ภายนอก
- บาง provider อาจ block iframe ด้วย CSP หรือ X-Frame-Options
- dashboard ที่เป็น private อาจต้อง login ซ้ำใน provider ปลายทาง
- ไม่ควรอนุญาต raw HTML embed โดยไม่มีการ sanitize หรือ policy ที่ชัดเจน

ผลลัพธ์ที่ควรได้:

- ลดความเสี่ยงจาก malicious embed
- มี policy ชัดเจนว่า provider ไหนใช้ได้
- ผู้ใช้เข้าใจเมื่อ dashboard เปิดใน portal ไม่ได้

## Phase 6 - Analytics And Maintenance

เป้าหมาย: ทำให้ระบบบริหารจัดการ dashboard ได้ต่อเนื่องในระยะยาว

ฟีเจอร์หลัก:

- Usage analytics
- Dashboard view count
- Last viewed date
- Last updated date
- Stale dashboard warning
- Owner reminder
- Dashboard review cycle
- Export inventory
- Admin report

ตัวอย่าง rule:

- ถ้า dashboard ไม่ถูกเปิดเกิน 90 วัน ให้ขึ้นสถานะ low usage
- ถ้า dashboard ไม่ถูก update เกิน 180 วัน ให้ owner review
- ถ้า dashboard ไม่มี owner ให้ admin ตรวจสอบ
- ถ้า embed เปิดไม่ได้หลายครั้ง ให้แจ้ง owner

ผลลัพธ์ที่ควรได้:

- dashboard ที่ไม่ถูกใช้งานถูกจัดการ
- dashboard ที่ล้าสมัยถูก review
- admin เห็นภาพรวมสุขภาพของ dashboard catalog

## Phase 7 - Advanced Capabilities

เป้าหมาย: เพิ่มความสามารถหลังจาก MVP และ governance หลักนิ่งแล้ว

ฟีเจอร์ที่อาจเพิ่ม:

- Request access
- Dashboard recommendation
- Duplicate detection
- AI search
- Natural language discovery
- Comment หรือ discussion ต่อ dashboard
- Subscription / notification
- Scheduled announcement
- Multi-tenant organization support
- API สำหรับ import dashboard จากระบบอื่น
- Integration กับ SSO หรือ directory service

ผลลัพธ์ที่ควรได้:

- ระบบเริ่มมีความสามารถแบบ internal analytics platform
- ผู้ใช้ค้นพบ dashboard ได้จากคำถามหรือบริบทงาน
- ลดการสร้าง dashboard ซ้ำซ้อน

## Suggested MVP Scope

ควรเริ่มจากขอบเขตนี้ก่อน:

- Authentication
- Role-based permission
- Admin category
- Manager sub-category
- Dashboard CRUD
- Provider type
- Embed URL + external URL fallback
- Dashboard metadata
- Publish/unpublish
- Search/filter
- Favorite
- Home sections: pinned, popular, recent, my team
- Basic audit log

ยังไม่ควรทำใน MVP:

- AI search
- Recommendation
- Duplicate detection
- Complex approval chain หลายชั้น
- Full marketplace rating
- Deep provider API integration
- Automated data freshness validation จาก provider ภายนอก

## First Technical Questions

คำถามที่ควรตอบก่อนเริ่มออกแบบ database และหน้าจอ:

- ระบบนี้มีผู้ใช้จากองค์กรเดียวหรือหลายองค์กร
- ใช้ authentication แบบไหน
- role มาจาก database เอง หรือผูกกับระบบเดิม
- category หลักจะมี owner ได้กี่คน
- project manager อยู่ได้หลาย category หรือไม่
- dashboard หนึ่งตัวอยู่ได้หลาย category หรือได้แค่ category เดียว
- ต้องการ approval ก่อน publish ตั้งแต่ MVP หรือทำใน phase ถัดไป
- provider แรกที่ต้องรองรับจริงคืออะไร
- ต้องจำกัด domain embed ตั้งแต่แรกหรือไม่
- ต้องเก็บ usage analytics ระดับใดในช่วงแรก
